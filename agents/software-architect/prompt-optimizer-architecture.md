## 3. 核心模块设计

### 3.1 Prompt Manager（提示词管理）

**职责**：管理提示词版本，支持历史回滚和差异对比

**数据结构**：
```python
class PromptVersion:
    id: str
    content: str           # 提示词内容
    version: int           # 版本号
    parent_id: str         # 父版本（便于追溯）
    created_at: datetime
    created_by: str        # "optimizer" 或具体用户
    metadata: dict         # 额外信息（场景、标签等）
```

**API 能力**：
- `POST /prompts` - 创建新提示词
- `GET /prompts/{id}/versions` - 获取版本历史
- `POST /prompts/{id}/compare` - 对比两个版本差异
- `POST /prompts/{id}/rollback` - 回滚到指定版本

---

### 3.2 Test Data Manager（测试数据管理）

**职责**：管理测试用例，支持分类、难例标注

**数据结构**：
```python
class TestCase:
    id: str
    input: str             # 测试输入
    expected_output: str   # 期望输出(可选)
    category: str          # 分类：功能/边界/幻觉检测
    difficulty: str        # 难度：easy/medium/hard
    tags: List[str]        # 标签
```

**重要原则**：预留 **验证集**，避免在测试集上过度拟合

---

### 3.3 Evaluator（评估引擎）

**职责**：执行测试并计算得分

**评分维度**（可自定义权重）：
| 指标 | 说明 |
|------|------|
| 准确率 | 输出与期望的匹配程度 |
| 格式合规 | JSON/结构化输出是否规范 |
| 领域匹配 | 领域术语使用是否正确 |
| 幻觉检测 | 是否有事实性错误（可接入 fact-checking） |
| 长度控制 | 输出是否在合理范围内 |

**输出**：
```python
class EvaluationResult:
    test_case_id: str
    prompt_version: int
    output: str
    scores: Dict[str, float]    # 各维度得分
    total_score: float          # 加权总分
    error_type: str            # 错误类型（可选）
```

---

### 3.4 Optimizer Core（优化核心）

**核心循环**：
```
while iteration < max_iterations:
    1. 基于当前版本生成 N 个变体
    2. 对每个变体运行评估
    3. 选择得分最高的进入下一轮
    4. 检查是否收敛（得分提升 < threshold）
```

#### 变异策略

| 策略 | 说明 | 示例 |
|------|------|------|
| **随机替换** | 随机替换词汇 | "详细" → "全面" |
| **语义润色** | 让 LLM 优化表达 | "帮我写" → "请生成" |
| **结构重组** | 调整段落顺序 | 把示例放到前面 |
| **Few-shot 调整** | 增删/修改示例 | 添加更多正例 |
| **约束强化** | 增加约束条件 | 添加 "必须" |

#### 搜索策略

| 策略 | 适用场景 | 优点 |
|------|----------|------|
| **随机搜索** | 初期快速探索 | 简单，不易陷入局部最优 |
| **贝叶斯优化** | 评估成本高 | 用少量尝试找到最优 |
| **进化算法** | 复杂非线性 | 并行搜索，全局寻优能力强 |

### 3.5 Workflow Scheduler（工作流调度）

**功能**：
- 长时间任务持久化（重启不丢）
- 并发控制（避免 API 限流）
- 失败自动重试 + 告警
- 定时任务（如每日自动优化）

**推荐**：Celery + Redis（轻量）或 Temporal（重任务）

## 5. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| **成本失控** | 迭代耗_tokens，费用暴涨 | 设置单任务预算、早停策略、缓存 |
| **过拟合** | 测试集高分，线上拉胯 | 预留验证集，周期用新数据验证 |
| **无限循环** | 不收敛，浪费资源 | 最大迭代次数 + 收敛阈值 |
| **模型幻觉** | 评估结果不可靠 | 引入人工抽检 + 多模型交叉验证 |

## 6. 快速验证路径

```
Phase 1 (1-2周): 核心逻辑
├── Python 脚本实现变异 + 评估循环
├── 本地 JSON 文件存储版本

Phase 2 (2-3周): 工程化
├── FastAPI 封装 API
├── PostgreSQL 持久化
├── Celery 任务队列
└── 基础前端（版本列表 + 评估结果）

Phase 3 (持续): 增强
├── 搜索策略优化（贝叶斯/进化）
├── 多模型支持
├── Webhook 集成
└── 监控告警
```

## 7. 总结

这个系统的本质是 **"提示词的 AutoML"**：

1. **核心循环**：变异 → 测试 → 评估 → 选择 → 变异
2. **关键模块**：Prompt Manager、Test Data、Evaluator、Optimizer
3. **技术栈**：Python + FastAPI + PostgreSQL + Celery
4. **起步**：先跑通最小闭环，再逐步工程化

> *简单、实用、安全，是架构的三大原则。* 🏗️
